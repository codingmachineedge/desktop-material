# Desktop Material — Feature Expansion Plan

## Context

`desktop-material` is a fork of GitHub Desktop (Electron + React + TypeScript + Sass) being rebuilt around Material Design 3. `MATERIAL_REDESIGN.md` is the implementation contract. The user supplied `Material Design UI Recreation.zip`; M3 was adapted from its `Desktop Material v2.dc.html` Settings History side sheet, while M18 remains the later full-app alignment pass.

The user wants a large feature expansion:

1. **Multi-account support** — largely exists already (recent commits: same-host multi-account sign-in, multi-account settings, account↔repository binding). Extend as needed for per-account data below.
2. **Browser-like tabs** — each tab = an open repository (user decision). Tabs are renameable and per-tab title styling is customizable: font size, color, font family/style, italic, bold, alignment, "and more".
3. **Per-account persistence in a local git repo** — tabs + settings saved per account; every touch of tabs/settings commits to a local git repo (one per account) under Electron `userData`; plus a **full undo history manager** UI built on that commit history.
4. **Multi-clone** — checkboxes on the account repository list in the clone dialog to select many repos; clone them in parallel or one-by-one (user's choice).
5. **Search upgrades everywhere** — every search bar gets: filters, regex search mode, and a **full regex builder** UI.
6. **One-click commit + push button** — commit message written by the existing Copilot integration.
7. **Scheduled auto commit + push** — periodic timer; global default + per-repo override (user decision).
8. **Auto git pull** — periodic; same scoping.
9. **Auto-merge all branches into main** — then delete merged branches, then push. Conflicts: **auto-resolve with the existing Copilot conflict-resolution engine**, skip branch only if Copilot fails (user decision).
10. **Same for worktrees** — merge worktree branches into main, delete worktrees after successful merge.
11. **Repo export/import** — export cloned repos to a file containing **only remote URLs**; import on another machine auto-clones them (reuses multi-clone engine).
12. **Built-in MCP server + "cheap" fallback** (user decision) — app hosts an MCP server exposing app control (repos/accounts/tabs/clone/commit/push/pull/branches/automation) to AI agents; fallback for non-MCP agents = same commands over a local HTTP JSON endpoint + tiny CLI. Localhost-only, token-gated, opt-in.
13. **Dynamic UI scaling** (user decision) — Preferences slider 50–200% + Ctrl+=/−/0, persisted per account, plus auto-fit on window/DPI change (toggle, composes with manual zoom, clamped).
14. **GitHub organization support** (user decision) — browse/clone full org repo lists in the clone dialog (with multi-clone checkboxes), org filter/grouping, publish new repos into an org.
15. **GitHub Actions panel** (user decision) — per-repo view of workflow runs (status/branch/event), re-run + rerun-failed, trigger workflow_dispatch with inputs, and in-app job log viewer.
16. **Desktop Plus parity** (user request) — absorb ALL features of https://github.com/desktop-plus/desktop-plus (MIT, fork of same upstream, v3.6.2): GitLab+Bitbucket integration (clone/PR/status), multi-window (Ctrl+Alt+N), commit search (title/message/tag/hash), commit graph visualization, merge-commit font styling, relative/absolute dates, delete pushed commits/tags, pull-all button, branch name in repo list, custom repo grouping, repo pinning, local-only branch warnings, custom default branch, branch sort options, multiple stashes per branch, permanent discard, SVG diff preview, distinct button icons, red destructive buttons, git identity display, branch push status indicators, per-repo editor override, branch-name preset scripts, hide recent repos, telemetry disabled, CLI. Strategy: study/port from desktop-plus source (MIT) where its diffs adapt cleanly, reimplement in material style otherwise.
    - **16a. Self-hosted GitLab login** (user request): sign-in flow accepts arbitrary GitLab server URLs (mirroring the GitHub Enterprise endpoint-entry flow), personal-access-token auth (works on all self-hosted instances; OAuth optional later), integrated with the multi-account model (`endpoint#id` account keys → several accounts per GitLab host, repo→account binding, per-account profiles) and with GitLab clone/PR/status features from the desktop-plus port.
17. **Non-blocking in-app dialogs** (user request) — dialogs render as non-modal in-app surfaces (floating panels/side sheets) instead of full modal dialogs, so the main window stays interactive while a dialog is open. OS-native pickers (file open/save) stay native.
18. **Notification centre** (user request) — in-app notification centre (bell + panel) collecting app events (clones, auto-commits, merge results, errors, PR/CI events from the existing notifications store); backed by **its own local git repo** where every action is committed: new notification, delete, edit, mark read, mark unread; plus a **notification history manager** UI to browse/restore that history (generalize the settings-history design).
19. **Workflow requirement** — commit + push to `origin/main` at every milestone (user decision); capture screenshots and publish to README, GitHub Pages, and the GitHub wiki as features land.

Renderer-side (dugite precedent). (2) `RepositoryTabsStore` — in-memory tab state persisted via ProfileStore. (3) UI: `RepositoryTabStrip` above toolbar, `TabStyleEditor` popover, `SettingsHistoryDialog` (menu `show-settings-history`, `CmdOrCtrl+Alt+Z`).

**profile-settings-registry.ts**: ALLOWLIST of localStorage keys (widths, confirmation/appearance/diff prefs from app-store.ts:417-520 constants) → `captureSettingsSnapshot/applySettingsSnapshot/describeSettingsChange`.
